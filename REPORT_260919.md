# Throughput Benchmark — 2026-06-19

Hardware: AMD Ryzen AI Max Pro 395+ (Strix Halo), Radeon 8060S iGPU  
Build: clang-17 / libc++ / ROCm + Vulkan backends  
Prompt: 256 output tokens, 3 runs after 1 warm-up, `temperature=0 seed=42`

## Results

| Model | Config | Backend | Run 1 | Run 2 | Run 3 | Avg t/s |
|-------|--------|---------|------:|------:|------:|--------:|
| DiffusionGemma 26B Q8_0 | diffusion | ROCm | 25.67 | 29.59 | 29.74 | **28.33** |
| DiffusionGemma 26B Q8_0 | diffusion | Vulkan | 29.21 | 29.43 | 28.28 | **28.97** |
| Gemma-4 26B MoE Q8_0 | gemma-mtp | ROCm | 53.91 | 52.13 | 52.19 | **52.74** |
| Gemma-4 26B MoE Q8_0 | gemma-mtp | Vulkan | 56.43 | 62.04 | 60.13 | **59.53** |
| Gemma-4 26B MoE Q8_0 | default | ROCm | 37.19 | 36.95 | 36.97 | **37.03** |
| Gemma-4 26B MoE Q8_0 | default | Vulkan | 41.19 | 40.86 | 40.93 | **40.99** |
| Qwen3-Coder-Next Q8_0 (~80B) | default | Vulkan only | 35.61 | 36.26 | 36.30 | **36.06** |

### Observations

- **Vulkan beats ROCm by 2–13%** across all tested configurations on the Strix Halo iGPU. The gap is largest on the MTP path (+13%), possibly due to better Vulkan shader scheduling on RDNA iGPU.
- **gemma-mtp vs default** (same model, same backend): speculative MTP drafting yields +42–45% throughput (52–60 t/s vs 37–41 t/s). The 3-draft-token speculative path is working correctly.
- **Vulkan diffusion sampling matches ROCm** (~29 t/s): the new Vulkan compute shader produces equal performance to the HIP implementation at this canvas size.
- **Qwen3-Coder-Next (~80B) at 36 t/s** on Vulkan is comparable to default Gemma-4 path, despite being 3× larger — the iGPU's unified memory bandwidth scales well with model size in memory-bound decode.

## Recent Custom Commits

### `d97f1fc6d` — vulkan: add Vulkan compute shader for DiffusionGemma entropy-bound sampling
*2026-06-19*

Added a GLSL compute shader (`diffusion_sample.comp`) so DiffusionGemma's multinomial sampling step runs on the Vulkan device instead of copying ~268 MB of logits to the host per step. The shader runs one workgroup (256 threads) per canvas token through three phases: parallel max reduction → argmax, Z normalization + entropy accumulation, and uniform-draw multinomial via prefix scan + serial walk. The backend dispatch is wired into `ggml-vulkan.cpp` and exposed via the shared `ggml_backend_cuda_diffusion_sample` proc-address name so `diffusion-gemma.cpp` can reach it without a new link-time dependency.

`diffusion-gemma.cpp` also gained buffer-type verification: before calling a backend's sampling function, it confirms the logits buffer actually lives on a device from that backend. This prevents the ROCm backend (registered even with `HIP_VISIBLE_DEVICES=""`) from crashing via `hipGetDevice()` when the model is running Vulkan-only.

### `d6cf0b288` — diffusion: device-side sampling reductions (upstream, ~1.4×/step)
*2026-06-11, upstream contributor: danielhanchen*

Moved argmax/entropy/multinomial reductions onto the GPU via a new `ggml-cuda` kernel, eliminating the ~268 MB/step host copy of canvas logits. The kernel is reached through the `ggml_backend_cuda_diffusion_sample` proc-address boundary (no new llama↔cuda link-time dependency). Falls back to the host path on non-CUDA or multi-GPU configurations. Results are bit-identical on greedy decode; stochastic output matches on all tested prompts (parallel reduction reorder affects entropy by ~1e-4, within the same FP-equivalence class as `--diffusion-kv-cache`). ### `ef5e2dcce` — diffusion-gemma: chunked causal prefill + cap encode outputs to canvas
*2026-06-17, upstream contributor: danielhanchen / potto007*

Fixed a crash on long prompts: the single-shot non-causal encode built an `[n_tokens, n_vocab]` fp32 logits tensor that overflowed 32-bit CUDA softcap indices past ~12k tokens (`n_head × N² > 2^31`). The prefill now runs in `n_ubatch`-sized causal chunks into a grow-only K/V store, and `llama-context` honors `n_outputs_max` to copy only the flagged (canvas) output rows instead of the full logits buffer. The per-turn compute buffer is now flat ~566 MiB regardless of context length; output is byte-identical when the prompt fits one ubatch, and 60k+ token prompts work where the single-shot encode crashed.

### `323f52cde` — ggml-cpu: pre-compute fused-op table at graph-compute time
*2026-05-20*

Previously `ggml_cpu_try_fuse_ops` ran inside the hot compute loop — called by every thread for every node, redundantly re-detecting the same static fusion candidates on each forward pass. This adds a `fused` array to `ggml_cplan` (allocated in the `work_data` tail): a single-threaded pre-pass at `ggml_graph_compute` time walks the graph and writes per-node skip-counts, so the per-thread inner loop does an O(1) array read instead of re-running the detection comparisons. Reduces fusion overhead from `O(n_threads × n_nodes)` per forward pass to `O(n_nodes)` once.

### `13af405d6` — ggml-cpu: enable threading for GET_ROWS/SET_ROWS when source is host-resident
*2026-05-20*

`GET_ROWS` and `SET_ROWS` were unconditionally single-threaded (n_tasks=1) with a FIXME noting that thread launch overhead dominates when GPU offloading is active. This change uses `n_tasks = n_threads` when the source tensor's buffer is host-resident and `n_tasks = 1` otherwise, so CPU-only embedding lookups can use the thread pool without adding latency to GPU-offloaded workloads.

### `2f3b668f2` — llama-graph: n_seqs_unq-based Q/K/V splitting for no-cache encoder path
*2026-05-20*

When `equal_seqs` is true the token layout is seq-major (all tokens of seq 0, then seq 1, …) so each sequence is an independent attention stream. The no-cache attention mask and K/V tensors are reshaped from a single `[n_tokens, n_tokens]` square into `n_seqs_unq` independent `[n_tps, n_tps]` blocks, matching the layout already used in the KV-cache path. This lets `build_attn_mha`'s existing multi-stream K logic handle the no-cache encoder, eliminates the O(n_tokens²) cross-sequence mask allocation, and allows the encoder to process batched sequences without redundant cross-sequence masking.
